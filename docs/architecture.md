# Architecture for PunchDeck

PunchDeck is a CLI tool that helps comedians store, organize, and retrieve their jokes, allowing them to ask for material by topic, tone, or constraints and build sets more easily. It supports:
* Querying jokes using natural language
* Building sets of a specified duration based on tone, topic, and rating
* Structured storage of joke data
* Optional LLM-assisted set generation

## Goals 

This section lists the goals of the project, organized into development tiers. 

### Development Tier 1 - Core Functionality

For core functionality, PunchDeck can:
* Ingest a `.xlsx` file, normalize its contents, and store jokes in a vector database with structured metadata.
* Require a single worksheet and a defined set of columns, with `joke` as the only required field.
* Allow users to query jokes using natural language.
* Retrieve relevant jokes using semantic search (embeddings) combined with structured filtering.
* Allow users to add new jokes or update existing jokes via file ingestion.
* Use `joke_id` to identify and update existing jokes when present.
* When `joke_id` is absent, detect duplicates using exact string matching on the `joke` field.
* Allow users to delete jokes from the dataset.
* Construct sets of a target duration by selecting jokes that meet specified constraints.
* Export query or set results to a standardized `.xlsx` format.

### Development Tier 2 - Enhanced Intelligence

For enhanced intelligence, PunchDeck can:
* Estimate the duration of jokes missing `duration_seconds` and store the result in `estimated_duration_seconds`.
* Provide an AI-assisted query mode via `ask --ai`, which:
  * Retrieves relevant jokes based on the query
  * Selects an appropriate subset based on constraints such as duration, tone, topic, and rating
  * Orders the selected jokes into a coherent set
  * Respects the requested duration target when possible
* Require users to supply their own API key for LLM-based features.

## Core Concepts 

## Data Model (Schema)

Each joke is stored as a structured record with the following fields:

* `joke_id` - A unique identifier generated by the system for each joke. Used to update existing jokes when provided.
* `joke` - The text of the joke. This field is required.
* `title` - (Optional) A brief title describing the joke.
* `duration_seconds` - (Optional) The confirmed duration of the joke in seconds, provided by the user.
* `estimated_duration_seconds` - (Optional) A system-generated estimate of the joke’s duration, based on existing jokes with known durations. This field is only populated when `duration_seconds` is not provided, and when the user explicitly requests estimation. Estimation may be unreliable when insufficient reference data is available, and the system tells the user when this is the case before performing the action. 
* `rating` - (Optional) A user-defined rating of the joke’s effectiveness, using an integer from 1 to 10.
* `topics` - (Optional) A comma-separated list of topics associated with the joke (e.g., "dating, work"). Internally stored as a list.
* `tone` - (Optional) A comma-separated list of tones describing the joke (e.g., "dry, absurd, dark"). Internally stored as a list.
* `notes` - (Optional) Free-form text for performance notes, such as delivery style, props, or reminders.

## Core Actions 

### Data Ingestion and Management

* `add` - Ingest an `.xlsx` file, normalize the data, and insert or update jokes in the database. If a `joke_id` is provided, the corresponding joke is updated. If no `joke_id` is provided, the system checks for an exact match on the `joke` field; if none is found, a new joke is created.
* `delete` - Remove a joke from the database using `joke_id`.

### Query and Retrieval 

* `ask` - Query jokes using natural language and return relevant matches based on semantic search (embeddings) combined with structured filtering.
* `ask --ai` - Query jokes and use an LLM to select, organize, and order a subset of the retrieved results into a structured set that satisfies the requested constraints.

### Data Output 

* `export` - Export query or set results to a standardized `.xlsx` file.

### Intelligence

* `estimate` - Generate `estimated_duration_seconds` for jokes missing `duration_seconds`. Estimation is only performed when explicitly requested by the user. If insufficient reference data is available, the system warns the user. If estimation proceeds under these conditions, the system falls back to a heuristic based on joke length (e.g., word count).

## System Behavior and Rules

PunchDeck follows a set of rules to ensure consistency and predictability:

* `joke` is the only required field for ingestion.
* All records are normalized into a consistent internal schema regardless of input completeness.
* `joke_id` is used to identify and update existing jokes when provided.
* When `joke_id` is not provided, duplicate detection is performed using exact string matching on the `joke` field.
* System-generated values (e.g., `estimated_duration_seconds`) never overwrite user-provided values.
* Estimation is only performed when explicitly requested by the user.
* If insufficient data exists for reliable estimation, the system warns the user before proceeding.
* Topics and tone values are provided as comma-separated strings and stored internally as lists.
* LLM-based features require a user-provided API key and are optional.
* If no API key is provided, LLM features fail gracefully with a clear message and do not affect core functionality.

## Input Format (Spreadsheet Contract)

PunchDeck expects input data in a standardized `.xlsx` format.

* The file must contain exactly one worksheet.
* The worksheet must include a `joke` column.
* All other columns are optional but must match the expected field names if provided.
* Supported columns include:
  * `joke_id`
  * `joke`
  * `title`
  * `duration_seconds`
  * `rating`
  * `topics`
  * `tone`
  * `notes`
* Topics and tone values should be provided as comma-separated strings.

## RAG / Retrieval Approach 

PunchDeck uses a retrieval-based AI approach to support natural-language querying over a joke library. Each joke is embedded and stored in a vector database so that semantically similar jokes can be found even when the query does not use exact keyword matches. Structured metadata such as topic, tone, rating, and duration is used alongside semantic retrieval to improve result quality. In AI-assisted mode (ask --ai), retrieved jokes are passed to an LLM, which selects and orders a subset into a structured set based on the user’s constraints.

## Design Principles

PunchDeck is designed with the following principles:

* **Simplicity over flexibility** - The system favors clear, predictable behavior over supporting many edge cases or input formats.
* **Ease of use** - The tool is designed to be approachable for non-technical users, with minimal setup and intuitive commands.
* **Structured data as a foundation** - Joke data is stored in a consistent schema to support reliable querying and filtering.
* **User data is the source of truth** - System-generated values (such as estimated duration) never overwrite user-provided data.
* **Explicit over implicit behavior** - Actions such as estimation or AI-assisted features are only performed when explicitly requested by the user.
* **Composability of features** - Core functionality (retrieval) is separated from enhanced functionality (LLM-based generation), allowing features to build on each other cleanly.
